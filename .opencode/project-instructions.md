# Распределение MCP-инструментов — smart-completions

Документ описывает, как агент **использует и распределяет MCP-сервера** в репозитории:
`/home/ramwsh/my-nix-cfg/generic-home-files/packages/pkgs-theia-ide/smart-completions`

Правила кодирования и производительного кода — `docs/rules.md` (обязательны и каноничны).

## Базовые принципы

1. **Использовать все сервера.** Каждый из 14 серверов имеет свою зону ответственности.
   Не игнорировать сервер из-за того, что задачу «можно» решить иначе.
2. **Строго предпочитать MCP.** Fallback на shell или сторонние средства допустим только
   тогда, когда нужное действие нельзя корректно или полноценно выполнить через MCP
   (например, прогон тестов, линтеров, typecheck, build — для них MCP-сервера нет).
3. **shell `grep` запрещён**, если доступны `xgrep` или `ripgrep`. Всегда предпочитать их.
4. **Веб по умолчанию через `lightpanda`.** `fetch` — только fallback.
5. **Детерминизм.** Один тип задачи → один предсказуемый инструмент. Не «перебирать»
   сервера наугад; идти по дереву выбора ниже.
6. Характер агента: точный, детерминированный, профессиональный, краткий, ясный.
   Не делать лишнего, следовать плану и явным указаниям пользователя.

## Поиск и навигация по коду

Дерево выбора (идти сверху вниз, брать первый подходящий пункт):

### 1. Точная семантика символа → `mcpls` (→ fallback `tree-sitter`)

Когда нужно достоверно: где **определён** символ, **кто его вызывает** (references),
какой **тип**, hover-документация, **диагностика** (реальные ошибки/warnings компилятора),
безопасный workspace-wide **rename**.

`mcpls` — мост LSP→MCP, опирается на языковой сервер TypeScript, поэтому даёт самые
достоверные ответы по TS-коду (не эвристику).

Примеры:
- где определён `HybridRetriever` и какой у него тип
- все места использования `EmbeddingIndexServiceImpl`
- тип возвращаемого значения метода `SweepBackendService`
- диагностика по `sweep-backend-service.ts` после правки

Fallback: если `mcpls` не поднялся или не дал результат → `tree-sitter`.

### 2. Структурные паттерны → `tree-sitter`

Когда нужен структурный анализ AST: найти классы/интерфейсы/декораторы/экспорты и
синтаксические паттерны, которые не привязаны к одному символу.

Примеры:
- классы, реализующие конкретный интерфейс
- все места `bind(X).toSelf().inSingletonScope()`
- все экспорты из `src/common/sweep/`

Инициализация один раз после старта: зарегистрировать проект tree-sitter по пути
`/home/ramwsh/my-nix-cfg/generic-home-files/packages/pkgs-theia-ide/smart-completions`.

### 3. Текст / строка / regex → `xgrep` (в проекте) или `ripgrep` (вне проекта)

- **`xgrep` (`xg serve`)** — основной текстовый поиск **по репозиторию проекта**.
  Индексированный, выгоден на повторных запросах по одному и тому же дереву.
  Инструменты: `search`, `find_definitions`, `read_file`, `index_status`, `build_index`.
- **`ripgrep`** — текстовый/regex-поиск **вне проекта** или разовый (one-off):
  другой путь, чужой репозиторий, `~/`, конфиги за пределами репозитория, быстрая
  свежая проверка без индекса.

shell `grep` использовать запрещено при наличии этих двух.

Примеры:
- найти все вхождения строки в `src/node/sweep/` → `xgrep`
- грубо проверить наличие подстроки в `~/.config` вне репозитория → `ripgrep`

> Примечание: `xgrep.find_definitions` — эвристика (regex), не AST. Для **точного**
> «где определён символ» предпочитать `mcpls` (п. 1), а не `xgrep`.

### 4. Чтение и запись файлов → `filesystem`

Канонический инструмент для чтения, создания и записи файлов проекта: исходники,
конфиги, тесты. **Все правки и записи идут через `filesystem`.**
`mcpls` остаётся read-only-интеллектом (определения/ссылки/типы/диагностика); из
write-операций через `mcpls` допустим только безопасный workspace-wide rename символа.

Примеры:
- прочитать `src/node/sweep/sweep-backend-service.ts`
- найти все `.ts` в `src/node/sweep/`
- записать изменения в файл

## Архитектура и качество

### `graphify` — высокоуровневый обзор связей

Когда нужно понять связи между модулями, зависимости компонентов и неочевидные
архитектурные связи. Использовать **для обзора** перед работой/рефакторингом.

Примеры:
- как связаны `SweepBackendService` и `EmbeddingIndexServiceImpl`
- от чего зависит `HybridRetriever`
- какие файлы ссылаются на `retrieval-queries.ts`

Если граф не построен или устарел — предварительно построить из корня репозитория.

### `code-review-graph` — проверка перед коммитом

Когда нужно оценить качество кода **перед коммитом**: dead code, большие/сложные
функции, граф зависимостей для ревью и рефакторинга.

Примеры:
- проверить `src/node/sweep/` на dead code
- найти большие или сложные функции
- понять граф зависимостей перед рефакторингом

Если граф не построен или устарел — предварительно построить из корня репозитория.

> Разведение: `graphify` — **обзор** связей на старте; `code-review-graph` —
> **гейт качества перед коммитом**.

## Веб и внешние источники

### `lightpanda` — основной веб-доступ (по умолчанию)

Когда: гуглинг / веб-поиск, открытие страниц, в т.ч. с JavaScript-рендерингом.
Headless-браузер с MCP (`goto`, `evaluate`, дамп страницы в markdown) — отдаёт
обработанный JS-контент в удобном для модели виде.

Примеры:
- найти в поиске актуальную информацию по теме
- открыть динамическую страницу (SPA) и получить контент
- прочитать страницу, где важен исполненный JS

### `fetch` — fallback веба

Когда: прямой URL без JS, либо когда `lightpanda` не справился/недоступен.
Не делать `fetch` основным путём — это запасной вариант после `lightpanda`.

Примеры:
- быстро прочитать статический URL документации
- fallback, если `lightpanda` упал на конкретной странице

### `mcp-deepwiki` — обзор и доки внешних GitHub-репо

Когда: нужен высокоуровневый обзор или документация по **внешнему GitHub-репозиторию**
(чужой код, библиотека, паттерн реализации).

Примеры:
- обзор архитектуры стороннего Theia-плагина
- понять, как устроен внешний репозиторий, прежде чем читать его исходники

Если в `deepwiki` нет данных по репозиторию → перейти на `lightpanda`/`fetch` по
самому репозиторию или его README.

### `git` — история, diff, status

Когда: история изменений, диффы между коммитами, статус репозитория и состав
изменений перед ревью или коммитом.

Примеры:
- что изменилось с конкретного коммита
- diff по файлу `sweep-backend-service.ts`
- список изменённых файлов

## Состояние, знания и рассуждение

### `memory` — первичный источник зафиксированного состояния

Когда: обращаться за **историей сессий** и зафиксированными фактами/решениями —
архитектурные решения, принятые соглашения, найденный баг, отложенная задача,
пометка «легаси, не трогать без причины», фиксированное значение конфигурации.

**Предпочитать `memory`, а не `obsidian`.**

### `mcp-obsidian` — локальная база знаний пользователя (вторично)

Когда: искать **устоявшуюся практическую информацию**, которая уже зафиксирована в
локальной базе знаний пользователя. Использовать **после** `memory`, как источник
наработанной практики, а не оперативного состояния сессий.

> Разведение: `memory` — оперативная история/решения сессий (приоритет);
> `obsidian` — устоявшаяся практическая база знаний (вторично).

### `sequential-thinking` — планирование сложных задач

Когда: **всегда на старте сложных/многошаговых задач** — планирование, рефакторинг,
разбор багов, реализация по плану. Для тривиальных задач не требуется.

## Таблица приоритета выбора инструмента

| Задача | Инструмент |
| --- | --- |
| Прочитать / записать / создать файл | `filesystem` |
| Точное определение / ссылки / тип / диагностика символа | `mcpls` (→ fallback `tree-sitter`) |
| Структурный паттерн: классы, декораторы, экспорты, реализации интерфейса | `tree-sitter` |
| Текст / строка / regex **по проекту** | `xgrep` |
| Текст / строка / regex **вне проекта** или разовый | `ripgrep` |
| shell `grep` | запрещён при наличии `xgrep`/`ripgrep` |
| Высокоуровневый обзор связей модулей | `graphify` |
| Качество / dead code / сложность **перед коммитом** | `code-review-graph` |
| История сессий и зафиксированные решения | `memory` (приоритет) |
| Устоявшаяся практика из локальной базы знаний | `mcp-obsidian` (после `memory`) |
| Гуглинг / веб-поиск / страницы с JS | `lightpanda` |
| Прямой URL без JS / fallback веба | `fetch` |
| Обзор / доки внешнего GitHub-репо | `mcp-deepwiki` |
| История git, diff, status | `git` |
| Планирование сложных / многошаговых задач | `sequential-thinking` |
| Прогон тестов / линтеров / typecheck / build | shell (нет MCP) |

## Типовой порядок работы над изменением

1. `sequential-thinking` — спланировать сложную/многошаговую задачу.
2. `memory` (→ при необходимости `obsidian`) — поднять зафиксированный контекст и решения.
3. `graphify` — обзор связей затрагиваемых модулей.
4. `mcpls` / `tree-sitter` / `xgrep` — точечная навигация и поиск по коду.
5. `filesystem` — внести правки.
6. `mcpls` диагностика + shell-прогон тестов, линтеров и typecheck для затронутой области
   (по `docs/rules.md`: п. 6 — тесты, п. 7 — линтеры и проверки типов).
7. `code-review-graph` — проверка качества перед коммитом.
8. `git` — diff/status и формирование коммита.
9. `memory` — зафиксировать важные решения для будущих сессий.
